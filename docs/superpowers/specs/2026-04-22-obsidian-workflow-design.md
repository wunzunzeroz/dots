# Obsidian Workflow Redesign — `~/brain`

## Purpose

Transform `~/brain` from an agent-readable knowledge base with sparse hand-written daily notes into a **strategic leadership ops layer** for Matt's Head of Engineering role at Sea-Flux. The system serves two readers — Matt (via Obsidian) and Claude Code (via `CLAUDE.md` router) — and must remain excellent for both.

**Primary goals:**
1. **Tighter daily loop** — daily note briefs Matt automatically, captures what he's doing today, and surfaces what's owed across projects/people/initiatives.
2. **Frictionless capture + AI-native** — any thought lands in the right place in under 3 seconds via Raycast; AI context (briefs, summaries) is available without leaving the daily loop.

**Non-goals:**
- Replacing Jira. This system is exclusively for strategic leadership work (drafting, deciding, reviewing, coordinating, prepping) — never execution-level tickets.
- Personal task management (shopping lists, errands, life admin). Todoist or similar can stay for that if needed.
- Mobile-first capture. Matt rarely captures on mobile; Raycast (desktop) is sufficient.

## Scope boundary (what lives here)

| Lives in brain | Lives in Jira |
|---|---|
| Drafting (RFCs, comms, updates) | Engineering tickets / bugs / features |
| Decisions & rationale | Team sprint execution |
| Stakeholder / cross-functional coordination | PR descriptions |
| Team 1:1s, hiring, performance | Story point estimates |
| Strategic initiatives & projects | Release/deploy steps |
| Matt's personal commitments as a leader | Team-level execution |

Projects *may* reference a Jira epic via frontmatter (`jira_epic: SF-1234`) as a link, but ticket state is Jira's job, not this system's.

## Task model

### Hierarchy: Action → Task → Project → Initiative

| Level | Definition | Example | Home |
|---|---|---|---|
| **Action** | Atomic, one-shot. Single keystroke/conversation/click. | "Email Greg re contract" | Today's daily note `## Actions`, OR nested under a Task/Project's sub-actions |
| **Task** | Multi-Action, known scope, no need for a file of its own | "Create Sentry data export dashboard" | `inbox.md` by default, OR inside a Project file, OR inside a person's file |
| **Project** | Multi-Task, outcome-focused, accumulating context/decisions | "V1 Turnoff" | `projects/<slug>.md` — its own file |
| **Initiative** | Multi-Project, strategic goal, multi-quarter | "Enable AI-first dev at Sea-Flux" | `initiatives/<slug>.md` — its own file |

### Rules

1. **Default home is the smallest viable one.** Action → daily note. Task → inbox. Don't create a Project file for a single atomic Task.
2. **Promote when it outgrows.** When a Task accumulates sub-actions, context, or decisions, promote to a Project file. When multiple Projects share a strategic arc, promote to an Initiative.
3. **One-way promotion.** Things grow up; they don't shrink back down.
4. **Every item has exactly one home file.** Dashboards *surface* via queries, they don't *store* duplicates.
5. **Capture must be cheap.** Enrichment (priority, date, linking) happens at review time, not capture time.

### Relationship to daily note

- **`## Priorities` section** lists 1–3 Tasks or Projects being **advanced today**. Each is a link to where it actually lives. Today-scoped sub-actions optionally nested beneath.
- **`## Actions` section** lists today-only atomic Actions that have no Task/Project home.

Priorities bleed across days (the underlying Task/Project persists). Actions do not (atomic — done, promoted, or killed).

## Views

### `overview.md` — command center (pinned, opens on launch)

Pure queries. No stored tasks. The "one place to see everything actionable."

Sections:
1. **Active initiatives** (Bases table: status, target, owner)
2. **🔥 Overdue & due today** (Tasks plugin query)
3. **🎯 Scheduled this week** (Tasks plugin query)
4. **🚀 Active projects** (Bases table with status, initiative, target, next_action)
5. **📥 Inbox — Tasks (no date)** (Tasks plugin query, grouped by priority)
6. **📥 Inbox — loose Actions** (Tasks plugin query, top-level actions in inbox.md)
7. **📒 Waiting on others** (Tasks plugin query for `#waiting` tag)

Noise control: every query has a `limit`, sorts by priority, filters `not done`, and excludes paused/done projects. Completed items drop out immediately.

### Daily note — today's working surface

See template in [Templates](#templates) section.

Rendering order:
1. Frontmatter (date, weekday)
2. `## Brief` — transcluded from `daily-briefs/<today>.md`
3. `## Priorities (1–3)` — links to Tasks/Projects being advanced today, with today-scoped sub-actions
4. `## Actions` — atomic one-shots for today
5. `## Notes` — free capture (meetings, thinking)
6. `## Dashboard` — today-filtered queries (due today, project next-ups, top inbox, waiting)
7. `## End-of-day close` — rollover checklist

## Capture mechanics

### Raycast (global, outside Obsidian)

Two Script Commands, each <30 lines of zsh:

| Hotkey | Name | What it does |
|---|---|---|
| `⌥A` | `brain-action` | Takes text arg, appends `- [ ] <text>` to today's daily note `## Actions` via Obsidian Advanced URI |
| `⌥I` | `brain-inbox` | Takes text arg, appends `- [ ] <text>` to `inbox.md` under `## Unprocessed` |

Captured items get an HTML-comment timestamp for forensic context: `<!-- captured 2026-04-22T14:15 -->`.

**Natural-language enrichment (optional, via text prefixes):**
- Priority: `!` → 🔺, `!!` → 🔼, `!!!` → 🔽
- Due date: `d:mon`, `d:2026-04-28`, `d:+3`
- Example: `!! d:+3 draft exec update` → `- [ ] draft exec update 🔼 📅 2026-04-25`

### Obsidian (inside the vault, picker-based)

Via QuickAdd plugin:

| Hotkey | Action | Destination |
|---|---|---|
| `⌘⇧A` | New Action (today) | Today's note `## Actions` |
| `⌘⇧I` | New Inbox item | `inbox.md` `## Unprocessed` |
| `⌘⇧P` | New Project Task | Picker → selected `projects/<slug>.md` `## Tasks` |
| `⌘⇧@` | New Person Task | Picker → selected `team/<person>/README.md` OR `team/<person>/1-1s/next-agenda.md` |
| `⌘⇧N` | New Project | Prompts for slug + initiative → creates `projects/<slug>.md` from template |
| `⌘⇧G` | Go to today | Opens `daily-notes/<today>.md` |
| `⌘⇧O` | Open overview | Opens `overview.md` |
| `⌘⇧B` | Open inbox | Opens `inbox.md` |

## Daily-brief integration

- `daily-brief` skill continues to write to `daily-briefs/YYYY-MM-DD.md` (no change to skill).
- Daily note template transcludes the brief at top: `![[daily-briefs/<% tp.date.now("YYYY-MM-DD") %>]]`
- Brief renders inline in the daily note; brief file stays independently editable and git-history-clean.
- Brief is scheduled via existing agent pattern (same as `standup-bot-agent`, `weekly-summary-agent`): **Claude Code trigger at 7:00am daily** runs `/daily-brief`.
- Raycast fallback: `⌥B` → runs `/daily-brief` on-demand (for days when the scheduled run misses or Matt starts late).

## Review cadence

Using Periodic Notes plugin for weekly/monthly/quarterly notes.

### Morning (5–10min)
1. Homepage opens `overview.md` or today's daily note (configurable — default: today's note).
2. Brief is transcluded at top (generated at 7am).
3. Scan brief, write 1–3 Priorities, pull items from inbox/projects into today's Actions as relevant.
4. Execute.

### Evening (3min)
1. Roll `## Actions` — per-task one-keystroke triage (roll to tmr / promote to inbox / kill).
2. Unfinished Priorities: append progress note, roll to tomorrow.
3. Any decisions made → `decisions/YYYY-MM-DD-<topic>.md`.

### Weekly review (Friday afternoon, ~30min)
- What moved (completed tasks in last 7d, grouped by source file — Dataview)
- What slipped (overdue tasks)
- Inbox triage — target: inbox at 0 by end
- Per-direct-report check
- Projects health
- Radar refresh (now inbox content)
- Next week's 3 priorities

### Monthly review (first Monday, ~60min)
- Initiative progress & blockers
- Project portfolio health — flag projects aged >60d with no movement
- Hiring pipeline
- Quarterly goal trajectory

### Quarterly review (start of Q, ~half day)
- Strategic alignment
- Initiative re-prioritization
- Team capacity / growth
- Big bets for Q

## Open questions (for implementation plan to resolve)

1. **Raycast → file append vs Advanced URI:** Direct file write is simpler; Advanced URI is more Obsidian-integrated (triggers plugin events like auto-reload, template variable expansion). Choose during implementation — likely direct write for speed.
2. **Prefix parsing in Raycast scripts:** Implement priority (`!`, `!!`) and date (`d:mon`, `d:+3`) shortcuts, or defer to v2? Initial stance: implement priority (trivial); defer date parsing unless quick.
3. **Daily note creation when appending at 10am to a non-existent file:** Raycast script must handle this gracefully — create from template if missing, then append.
4. **Homepage default:** Opens today's daily note (if Matt wants action-first) or `overview.md` (if Matt wants overview-first). Plan should default to today's daily note; expose config in plugin settings.
5. **End-of-day rollover one-keystroke hotkey:** Implement as QuickAdd macro or Templater user script. Needs spec during implementation.
6. **Migration of existing radar.md content to inbox.md:** Triage pass — some items become Project Tasks, some Actions, some scheduled. Explicit migration step in plan.
7. **Hiring / stakeholders / me / business / product / tech folders:** Remain as reference content. No Task storage changes there initially. Revisit once system is running.
